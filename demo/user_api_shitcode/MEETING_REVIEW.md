## 二、技术研讨议题

### 议题一：手写工具函数 vs 使用标准库

**背景**：
团队中存在"手写实现"的传统，这虽然体现了团队成员的技术热情，但也带来了一定的维护成本。

**讨论点**：
- 手写实现是否应该作为"团队技术积累"保留？
- 如何平衡"轮子再造"与"站在巨人肩膀上"的关系？
- 建议建立"手写函数库"文档，记录每个手写函数的适用场景

**建议解决方案**：
建议成立"代码规范化小组"，负责制定以下规范：
1. 什么情况下可以手写实现
2. 手写实现的代码审查标准
3. 手写函数的文档规范

**预计耗时**：2-3周调研，1周评审

---

### 议题二：SQL字符串拼接 vs ORM

**背景**：
当前代码中存在大量直接拼接 SQL 字符串的情况，虽然直观但存在安全风险和可维护性问题。

**正方观点**（@某同事）：
- SQL 字符串拼接更直观，性能更好
- 团队成员对 SQL 较熟悉，学习成本低
- 对于简单场景，ORM 的复杂度是不必要的

**反方观点**（@另一同事）：
- ORM 可以防止 SQL 注入
- ORM 更容易进行数据库迁移
- 符合"代码即文档"的理念

**建议解决方案**：
建议进行为期一个月的技术验证，对以下方案进行 POC：
1. SQLAlchemy
2. Tortoise ORM
3. SQLModel
4. 保持现状（加安全防护）

**预计耗时**：4-6周（含调研、验证、评审）

---

### 议题三：时间存储格式选型

**背景**：
当前代码中使用时间戳字符串存储时间，存在格式不统一的问题。

**业界方案调研**：

| 方案 | 代表项目 | 优点 | 缺点 |
|-----|---------|-----|------|
| 时间戳字符串 | 当前方案 | 简单直观 | 不可读、时区问题 |
| ISO 8601 | 阿里系 | 国际标准、可读性好 | 存储空间大 |
| UTC 时间戳 | 国际化项目 | 统一时区、计算方便 | 需要转换 |

**建议**：
建议成立"时间格式标准化小组"，对全站时间使用情况进行调研，形成统一规范。

**预计耗时**：2周调研 + 1周评审

---

### 议题四：API 响应格式统一

**背景**：
当前 API 响应格式不统一，有的返回 dict，有的返回 `JSONResponse`，有的手写 JSON 字符串。

**建议方案**：
建议统一为以下格式：
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

**影响**：
- 涉及所有接口的改造
- 需要前端团队配合
- 需要制定 API 规范文档

**建议**：
建议将此需求纳入下个季度的技术债务清理计划，单独作为一个 Epic 来管理。

**预计工时**：约 20 人天
